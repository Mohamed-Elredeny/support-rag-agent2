Are these results good or bad? Honest read

Solid methodology, moderate accuracy, one known & bounded weakness. The headline is the {{ '%.0f'|format(cv.accuracy * 100) }}% cross-validated routing accuracy — the honest out-of-sample number (every query scored by thresholds it never influenced), not the flattering {{ '%.0f'|format(ins.accuracy * 100) }}% in-sample figure.

  • Retrieval is perfect but not the real test. recall@1 100% / MRR 1.00 is expected over just 10 KB entries — almost any embedder would score this. The hard problem is knowing when not to answer.
  • The costliest error is controlled. Answerable queries are never missed (answer recall 100%), and only a single decline→answer slips through the router in isolation — a code request that the runtime LLM scope-guard catches before replying.
  • The real weakness: it over-answers ambiguous / near-scope queries. Under CV, clarify recall is {{ '%.0f'|format(clarify_recall * 100) }}% and decline recall {{ '%.0f'|format(decline_recall * 100) }}%. Cosine similarity alone can't tell “Where is my data?” (storage vs export) apart, so it answers instead of clarifying. Fixing this needs an intent/ambiguity signal beyond similarity — documented as future work.
